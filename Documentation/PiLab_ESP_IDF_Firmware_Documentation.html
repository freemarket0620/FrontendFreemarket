1. Project overview

This firmware turns an ESP32-P4 board into a small web-hosted PLC-style controller. It serves a Vue single-page application directly from firmware, exposes HTTP API endpoints for the UI, runs a deterministic PLC scan loop, compiles and executes AngelScript machine logic, manages a process image for I/O, persists tags in NVS, and stores user files in LittleFS.

Target

ESP32-P4 using ESP-IDF 6.x, Ethernet networking, LittleFS storage, and an embedded Vue production build.

Frontend

The old separate HTML pages have been replaced by one Vue SPA embedded into firmware as index.html, index.css, and app.js.

Runtime

A 1 ms hardware timer drives the PLC timing path. AngelScript scan() currently runs every 5th tick, giving a nominal 5 ms script scan period.

Storage

LittleFS is mounted at /littlefs for file management. PLC tags are persisted separately in NVS under the plc_tags namespace.

3. Build and flash process

Prerequisites

  • ESP-IDF 6.x command environment configured.
  • Target set to esp32p4.
  • The Vue production build copied into main/web/.
  • USB/JTAG or serial flashing connection to the ESP32-P4 board.

Normal build

idf.py set-target esp32p4
idf.py build
idf.py flash monitor

Clean rebuild

idf.py fullclean
idf.py build
idf.py flash monitor

Updating the Vue SPA

Build the Vue app on the PC, then copy the three production files into the ESP-IDF project.

# In the Vue project
npm install
npm run build

# Copy output into the ESP-IDF project
copy dist\index.html              ..\P4_PLC_Base\main\web\index.html
copy dist\assets\index.css        ..\P4_PLC_Base\main\web\assets\index.css
copy dist\assets\app.js           ..\P4_PLC_Base\main\web\assets\app.js

# Then rebuild firmware
idf.py build
idf.py flash monitor
Important: the firmware expects stable Vue output names: assets/app.js and assets/index.css. Keep the Vite config fixed-output setup, otherwise hashed filenames will require changing main/CMakeLists.txt and HTTP handlers.

4. Boot sequence

  1. app_main() logs startup.
  2. NVS is initialized. If NVS pages are invalid or version-mismatched, NVS is erased and re-initialized.
  3. LittleFS is mounted at /littlefs using partition label storage.
  4. Heap status is logged for 8-bit heap, internal heap, and PSRAM.
  5. ethernet_web_start() starts networking and the HTTP server.
  6. udp_server_start() starts the UDP test server.
  7. perf_test_start() starts PLC I/O, AngelScript runtime support, and the hardware-timer-driven scan loop.
extern "C" void app_main(void)
{
    nvs_flash_init();
    mount_littlefs();
    ethernet_web_start();
    udp_server_start();
    perf_test_start();
}

Adding a new API endpoint

  1. Add a static handler function in ethernet_web.c.
  2. Set the response content type correctly.
  3. Validate request body length before reading POST data.
  4. Register the handler in start_webserver().
  5. Make sure the route begins with /api/ so it cannot conflict with Vue SPA routes.

7. PLC runtime

The PLC timing path is implemented in perf_test.cpp. The current configuration is:

SettingValueMeaning
IO_TICK_PERIOD_US10001 ms base I/O tick.
SCRIPT_SCAN_DIVIDER5Run script every 5 ticks.
SCRIPT_SCAN_PERIOD_US5000Nominal 5 ms AngelScript scan period.
REPORT_PERIOD_MS5000Performance report interval.

Runtime cycle

  1. Hardware timer fires at 1 ms.
  2. PLC task receives notification.
  3. plc_io_tick_1ms() updates input/debounce/process-image timing.
  4. Every 5th tick, if PLC mode is RUN, script_engine_run_scan() executes the active AngelScript scan() function.
  5. plc_io_apply_outputs() applies output state.
  6. Performance statistics are collected and periodically logged.

RUN/STOP mode

perf_test_is_plc_running() and perf_test_set_plc_running() control the runtime state. The web API exposes this through /api/plc_mode. File upload and other storage-heavy operations should be kept disabled while the PLC is running to protect scan timing.

8. Tag system

The tag registry is implemented by plc_tags.cpp/.hpp. Tags are runtime variables that can be visible to the HMI, writable through the web UI, and optionally exposed to AngelScript.

Tag limits and fields

Symbol/FieldMeaning
PLC_TAG_MAX_COUNT = 64Maximum number of registered tags.
PLC_TAG_NAME_MAX = 32Maximum tag name buffer length.
PLC_TAG_DESC_MAX = 64Maximum description buffer length.
typebool, int, or float.
writableAllows writes from the HMI/API.
retentiveDesign intent for values that should survive restart.
hmi_visibleControls visibility to HMI tooling.
script_visibleControls AngelScript global registration.
min_value/max_value/unitsMetadata for UI validation/display.

Persistence

Tags are stored in NVS using namespace plc_tags and key tags_json. This is separate from LittleFS file storage.

Name rules

  • Names must start with a letter or underscore.
  • Names may contain letters, numbers, and underscores.
  • Names may not collide with reserved words such as scan, true, false, bool, int, float, control keywords, or built-in I/O names such as I0, Q0, AI0, AO0.

9. File storage

LittleFS is mounted at /littlefs using the partition label storage. The file manager API exposes list, upload, download, view, delete, and mkdir operations.

Latency rule: file uploads and directory creation should be disabled while the PLC is running. This matches the current design decision to avoid storage operations damaging PLC scan latency.

Recommended storage use

  • User scripts.
  • HMI layouts or JSON configuration.
  • Exported logs or captured diagnostic files.

Do not store the Vue app in LittleFS yet

The current best design is to keep the Vue SPA embedded in firmware. It is only three files and avoids LittleFS read-path overhead for the application shell. LittleFS should remain focused on user-managed files.

10. Extension guide

Adding a new Vue page

  1. Add the Vue route/component in the Vue source project.
  2. Rebuild with npm run build.
  3. Copy dist/index.html, dist/assets/index.css, and dist/assets/app.js to main/web.
  4. If the new page must support hard refresh at a new top-level path, register that path in ethernet_web.c to use spa_index_get_handler.
  5. Rebuild and flash firmware.

Adding a new frontend route on the firmware side

httpd_uri_t new_page_uri = {
    .uri = "/newpage",
    .method = HTTP_GET,
    .handler = spa_index_get_handler,
    .user_ctx = NULL
};
register_uri_checked(g_http_server, &new_page_uri);

Adding a new API route

static esp_err_t my_api_get_handler(httpd_req_t *req)
{
    httpd_resp_set_type(req, "application/json");
    return httpd_resp_sendstr(req, "{\"ok\":true}");
}

httpd_uri_t my_api_uri = {
    .uri = "/api/my_feature",
    .method = HTTP_GET,
    .handler = my_api_get_handler,
    .user_ctx = NULL
};
register_uri_checked(g_http_server, &my_api_uri);

Adding a new source file

Add the file to main/CMakeLists.txt under SRCS. If it needs a new ESP-IDF component, add that component under REQUIRES.

Changing scan timing

Edit the constants in perf_test.cpp. Be careful: lowering the script scan period increases CPU load and reduces headroom for web/API work.

static constexpr uint32_t IO_TICK_PERIOD_US = 1000;
static constexpr uint32_t SCRIPT_SCAN_DIVIDER = 5;

Adding physical I/O

The current process image abstracts I/O as arrays of DI, DO, AI, and AO values. Add physical GPIO or ADC handling inside plc_io.cpp, keeping the public interface stable where possible.

11. Test checklist

Firmware build

  • Pass: idf.py build completes with no linker errors.
  • Pass: idf.py flash monitor boots and logs LittleFS mount success.
  • Pass: monitor shows Ethernet/web server startup.

SPA routing

  • Hard refresh /.
  • Hard refresh /hmi.
  • Hard refresh /tags.
  • Hard refresh /files.
  • Hard refresh /script.
  • Hard refresh /editor.

Browser DevTools

  • No failed requests for /assets/app.js.
  • No failed requests for /assets/index.css.
  • No failed API calls for /api/command_center, /api/plc_data, /api/script_status, /api/tags, or /api/files/list.
  • No requests for removed old pages such as indexFiles.html, indexTags.html, index23_plc_data.html, or indexPrism_p4_api.html.

Functional tests

  • Command Center updates without duplicate polling storms.
  • HMI view survives route switching.
  • HMI button/write controls call /api/plc_write.
  • Tag page can read/write tag definitions.
  • Script page can upload, compile, and report errors with line numbers.
  • File Manager can list/view/download/delete files.
  • Upload and mkdir controls are disabled or rejected when PLC is running.
